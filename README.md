## Запуск

1. Скопировать .env.example в .env и заполнить;
2. docker compose -f docker-compose.yml up -d;
3. Зайти в контейнер с приложением (backend) и выполнить миграции ```./backend -s```.

## Запуск в режиме разработки
1. Скопировать .env.example в .env и заполнить;
2. docker compose -f docker-compose.yml run -p 5432:5432 db;
3. Запустить приложение, например, по F5 через VSCode.

## Доступные переменные окружения
* POSTGRES_DB - имя БД (используется контейнером постгрес)
* POSTGRES_USER - пользователь БД (используется контейнером постгрес)
* POSTGRES_PASSWORD - пароль БД (используется контейнером постгрес)
* GORATOR_DEBUG_USERS_PASSWORD - пароль для тестовых пользователей при запуске миграций, работает с установленнной в 1
переменной GORATOR_IS_DEBUG
* GORATOR_DB_HOSTNAME - хост постгреса
* GORATOR_DB_PORT - порт постгреса
* GORATOR_DB_USERNAME - имя пользователя для подключения к постгресу
* GORATOR_DB_PASSWORD - пароль для подключения к постгресу
* GORATOR_IS_DEBUG - флаг, обозначающий запуск в дебаг-режиме
* GORATOR_SKIP_AUTH_CHECK - флаг, показывающий, что для входящих запросов будет игнорироваться сессия из заголовков и
вместо соотнесения сессии с пользователем будет браться id пользователя из GORATOR_AUTH_USER_ID. Работает только с 
GORATOR_IS_DEBUG=1
* GORATOR_AUTH_USER_ID - id пользователя, от имени которого будут обрабатываться входящие запросы. Работает только при
установке переменных GORATOR_IS_DEBUG=1 и GORATOR_SKIP_AUTH_CHECK=1. Нужно, чтобы быстро слать определенные запросы
через curl и не получать перед этим сессию
* GORATOR_ALLOWED_ORIGIN - url, запросы с которого разрешено обрабатывать (CORS)
* GORATOR_SALT - соль для хеширования пароля в БД

## Варианты запуска приложения и отличия
1. Запуск как сервис (просто собрать и запустить приложение)
2. Запуск с ключом -s создаст все таблицы, связи и, при дебаг-режиме (GORATOR_IS_DEBUG=1), 
тестовые записи основных сущностей: организация, команда, пользователи с ролями админа и вьювера

## Быстро тестировать запросы через curl, игнорируя получение сессии
Например, у нас есть пользователь JustAnotherUser с id = 7 в БД. Тогда, чтобы все запросы приходили с якобы сессией от
этого пользователя необходимо установить следующие переменные окружения:
```
GORATOR_IS_DEBUG=1
GORATOR_SKIP_AUTH_CHECK=1
GORATOR_AUTH_USER_ID=7
```
После этого можно отправлять запросы, примеры которых есть в каталоге curl. Например:
```
cd curl/create
curl http://localhost:8080/user -d @user.json -X POST  -v
```
Или
```
cd curl/update
curl http://localhost:8080/user -d @user.json -X PUT  -v
```
